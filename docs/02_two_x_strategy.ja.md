# 現在のツールより2倍高速化する方法

## まず：ユニバーサルに2倍は不可能

ビルドツールはすべてのワークロードで2倍高速にはなり得ません。

反例：

```text
1. 何もしないビルドはすでに50ミリ秒
2. ビルドには避けられない10分間のリンカ処理がある
3. ターゲットは意図的に完全リリースビルド
4. キャッシュがコールド状態でリモートワーカーが利用できない
5. コンパイラがすべてのランタイムを支配し、並列化できない
```

したがって、正しい製品の主張は：

```text
小さな変更セットを伴う一般的なインクリメンタルモノレポCI/ローカルビルドにおいて2倍高速
```

これが実現可能です。

## 2倍への主な道筋

主なレバーは高速なコマンドの実行ではなく、実行するコマンドの数を減らすことです。

```text
スピードアップ = 旧作業量 / 新作業量
```

現在のツールが160のパーティションを再ビルドし、FrostBuildが9を再ビルドする場合：

```text
作業削減 = 160 / 9 ≈ 17.8倍
```

プランナーのオーバーヘッドがあっても、2倍は現実的です。

## 1. マイクロパーティションの剪定

現在のツールはしばしばプロジェクトやターゲットの粒度で止まっています。

FrostBuildはもっと細かくする必要があります：

```text
プロジェクト -> パッケージ -> モジュール -> シンボル/テストパーティション
```

メタデータカタログを利用：

```text
変更されたファイル
  -> 変更されたパーティション
  -> 逆依存関係のクロージャ
  -> 影響を受けるテスト
  -> キャッシュの検索
  -> 最終実行セット
```

これがSnowflakeスタイルのメタデータ剪定の導入ポイントです。

## 2. 安全なリグレッションテストの選択

テストはしばしばCIの支配的な部分です。

2つの情報源を利用：

```text
静的依存関係グラフ：
  インポート、クラス参照、パッケージ参照

動的依存関係グラフ：
  syscall/ファイルトレース、クラスローダートレース、カバレッジデータ
```

次に依存関係に基づいてテストを選択。

デフォルトルール：

```text
不確かな場合は、より多くのテストを実行
```

リスク管理：

```text
- 毎晩の完全テスト実行
- 定期的なランダムテストサンプリング
- フ flakyなテストを隔離
- 選択されたテストと完全実行の精度を比較
- 選択サービスが停止しても失敗しないようにし、ビルドツールのベースラインにフォールバック
```

## 3. 遅延出力のマテリアライゼーション

リモート実行はクライアントが多くのアーティファクトをダウンロードする場合遅くなる。

より良い方法：

```text
すべての出力をCASに保存
要求された出力のみをマテリアライズ
その他のメタデータは保持
```

これはBuck2の遅延マテリアライゼーションに似ています。

## 4. コストベースのスケジューラ

通常のスケジューラは準備完了したアクションのみを見る。

より良いスケジューラは次を考慮：

```text
- クリティカルパス
- アクションの履歴
- キャッシュの局所性
- ワーカーのツールチェーンの利用可能性
- 出力サイズ
- ネットワーク帯域幅
- 失敗率
```

次にクエリプランナーのようにアクションをスケジューリング。

```text
リモートキャッシュヒットの大きな出力：
  リモートに保持し、必要に応じてダウンロード

長時間のコンパイルとキャッシュミス：
  リモートワーカー

出力が小さく短いアクション：
  RPCよりローカルワーカーの方が速い場合も
```

## 5. 永続ワーカーとコンパイラデーモン

多くのビルドはコンパイラの起動に時間を浪費している。

利用例：

```text
- TypeScriptサーバ / tsserverのようなインクリメンタル状態
- Rustコンパイラキャッシュ / sccacheスタイルの動作
- Kotlinのインクリメンタルコンパイラ統合
- JVMの永続ワーカー
- 長寿命の言語ワーカー
```

Metaによると、KotlinモジュールはBuck2にKotlinインクリメンタルビルドを導入した後、最大3倍高速になったとのこと。これはコンパイラレベルのインクリメンタリズムがビルドグラフの剪定と積み重なる可能性を示しています。

## 6. Nixスタイルの環境スナップショット

ビルドキャッシュは環境もキーに含める必要があります。

```text
悪いキー：
  ソースハッシュ + コマンド

良いキー：
  ソースハッシュ + コマンド + ツールチェーンのクロージャ + プラットフォーム + 設定
```

Nixスタイルのクロージャハッシュは正確性とキャッシュ共有を向上させます。

## 7. 正確性を犠牲にしない動的依存関係

一部の依存関係は静的に見えません：

```text
コード生成
リフレクション
動的インポート
C/C++ヘッダの発見
Rustのbuild.rs
TypeScriptのパスエイリアス
設定ファイル
```

利用方法：

```text
- サンドボックス実行
- syscallトレース
- コンパイラのdepfile
- 動的依存関係の記録
```

その後、各アクション後にカタログを更新。

## 最も効果的な2倍の勝利要素

インパクト順：

```text
1. テスト選択
2. プロジェクトレベル以下のマイクロパーティション剪定
3. リモートキャッシュヒット率の向上
4. 遅延マテリアライゼーション
5. 永続コンパイラワーカー
6. リモート実行のスケジューリング
7. Zig/Rustを使ったエンジンのオーバーヘッド低減
```

## 最初にやるべきではないこと

すべてをZigに書き換えることから始めない。

まず、アルゴリズムの勝利を証明してください：

```text
プランナーは安全に選択された作業を50%以上削減できるか？
安全モードで必要なアクションを一つも見逃さないか？
実際のワークロードでCIのウォールタイムを短縮できるか？
```

その後、ホットなエンジンパスの書き換えを行います。
