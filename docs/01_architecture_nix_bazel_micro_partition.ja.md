# Nix + Bazel + マイクロパーティションアーキテクチャ

## コアアイデア

```text
Nixレイヤー:
  正確な環境とツールチェーンのクローズ

Bazel/Buckレイヤー:
  アクショングラフ、依存関係グラフ、リモートキャッシュ、リモート実行

スノーフレークレイヤー:
  メタデータカタログとパーティションプルーニング
```

新しいツールは単なるビルドランナーではありません。ビルドのクエリエンジンに近いものです。

```text
入力:
  git diff
  要求されたターゲット
  プラットフォーム
  環境
  キャッシュ状態

プランナー:
  影響を受けるパーティションのプルーニング
  依存関係の展開
  テストの選択
  キャッシュのルックアップ
  ワーカーの配置

エグゼキューター:
  ローカル/リモートの並列アクション
  CASアーティファクトストア
  遅延マテリアライゼーション
```

## なぜNixが重要か

Nixはパッケージを孤立してビルドし、未宣言の依存関係を避けます。これは正しいアクション環境に必要な正確な特性です。

すべてのビルドアクションには環境ハッシュを含めます。

```text
action_key = hash(
  コマンド,
  引数,
  宣言された入力,
  環境クローズハッシュ,
  プラットフォーム,
  選択された環境変数
)
```

これにより、マシン間の正確性が保証されます。

## なぜBazel/Buckが重要か

BazelとBuck2は、ビルドが小さなアクションのグラフであることを教えてくれます。

```text
ターゲット -> アクション -> 入力 -> 出力
```

高速なビルドシステムは次のことを知るべきです。

```text
- グラフを理解する
- 独立したアクションを並列で実行する
- アクションキーによる出力のキャッシュ
- ローカルCPUが不足する場合はリモートワーカーを使用
```

## なぜマイクロパーティションが重要か

スノーフレークのマイクロパーティションは、クエリエンジンが不要なデータをスキップできるメタデータを格納します。ビルドの対応は次の通りです。

```text
ソースファイル / モジュール / ターゲット / テスト = マイクロパーティション
メタデータ = インポート、エクスポート、ハッシュ、逆依存関係、テストカバレッジ、コスト
プルーニング = 関係のないビルド/テスト作業をスキップ
```

例：

```text
変更点:
  packages/ui/Button.tsx

通常のプロジェクトレベルの影響:
  packages/ui
  apps/web
  多くのWebテスト

マイクロパーティションの影響:
  Buttonパーティション
  Buttonをインポートするページ
  Buttonをカバーするテスト
```

ローカルプロトタイプ：

```text

```

本番環境：

```text
メタデータカタログ：Postgres / FoundationDB / SQLiteのレプリケーション

リモート実行：REAPI互換のワーカー
ローカル状態：RocksDB / SQLite
```

## クエリプランナーのアナロジー

SQLクエリエンジン：

```text
述語 -> パーティションプルーニング -> 実行計画 -> ワーカー
```

ビルドエンジン：

```text
git diff + ターゲット -> ビルドパーティションプルーニング -> アクションプラン -> ワーカー
```

これが通常のビルドツールとの主な設計上の違いです。

## 正確性ルール

次のいずれかが成り立たない限り、作業をスキップしてはいけません。

```text
1. パーティションが依存関係のメタデータによって影響を受けていないことが証明されている
2. 正確なアクションキーにキャッシュヒットがある
3. ユーザーが確率的テスト選択のリスクを明示的に受け入れている
```

本番環境では、デフォルトモードは保守的であるべきです。

```text
セーフモード:
  影響を受ける作業を過剰に選択
  影響を受けない作業を決して選択しない

高速モード:
  確率的テスト選択を使用
  毎晩/完全な検証を要求
```
